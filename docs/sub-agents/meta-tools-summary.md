# Meta Tools Summary

**Date:** 2026-04-25
**Purpose:** Decide how this repo should build Meta Graph / Marketing API tools for the weekend MaaS MVP.

## Bottom Line

Do **not** adopt an MCP server as the primary integration layer for this repo.

For the weekend MVP, the lowest-risk path is:

1. Build a **thin server-side TypeScript Meta client** around a small set of explicit REST calls.
2. Validate every request/response with local schemas.
3. Use Meta's **official Postman workspace** as the request reference and test fixture source.
4. Keep writes limited to **paused draft creation only**.

If the later CrewAI worker is Python-based, the **official Python Business SDK** is acceptable inside that worker. For this repo's first MVP tools, direct REST is still the cleaner choice.

## MCP Server Verdict

I found **community** Meta Marketing API MCP servers on GitHub, but I did **not** find an official Meta-maintained MCP server in Meta's official docs, official Postman workspace, or official SDK materials.

Recommendation:

- **Do not rely on community MCP for the weekend MVP.**
- Reason: it adds an extra abstraction layer, another dependency surface, and another place for token/config mistakes.
- Reason: the weekend goal is deterministic app tools, not natural-language access to Meta from the editor.

The strongest community example I found is `brijr/meta-mcp`, but it is still third-party software with broad ad-management powers. That makes it unsuitable as the foundation for a high-confidence MVP where we care about safety and minimal moving parts.

## Recommended Implementation Approach

### Preferred path for this repo

Use direct HTTPS calls from server-side TypeScript:

- `GET /me/adaccounts`
- `GET /act_{account_id}/insights`
- `GET /act_{account_id}/campaigns`
- `POST /act_{account_id}/campaigns`
- `POST /act_{account_id}/adsets`
- `POST /act_{account_id}/adcreatives`
- `POST /act_{account_id}/ads`

Build a small internal client such as:

- `listAccessibleAdAccounts(token)`
- `getAccountInsights(accountId, params)`
- `createPausedCampaignDraft(accountId, payload)`
- `createPausedAdSetDraft(accountId, payload)`
- `createCreativeDraft(accountId, payload)`
- `createAdDraft(accountId, payload)`

### Why this is the best weekend path

- Matches the repo's existing `Next.js + TypeScript` stack.
- Easier to log, sanitize, trace, and debug than an MCP bridge.
- Easier to enforce the safety rule: **no live changes**.
- Lower hallucination risk because each tool maps to one known endpoint and one narrow payload shape.
- Easier to prove to judges because the flow is explicit.

### Low-hallucination coding pattern

For each tool:

1. Start from one official Postman example or official SDK example.
2. Encode only the fields needed for MVP.
3. Validate inputs with local schemas before sending.
4. Validate Meta responses before passing them to agents.
5. Log sanitized request metadata, response status, and object IDs.
6. Fail closed on unknown write fields.

## SDK / Library Recommendation

### Primary recommendation

For the app layer: **use native `fetch` with a tiny typed wrapper**, not a heavy SDK-first design.

Reason:

- Fewer abstractions for a very small endpoint set.
- Better control over exactly what is sent to Meta.
- Easier to mirror official Postman requests.
- Easier to keep the write surface narrow.

### Secondary recommendation

If a Python CrewAI worker becomes necessary, use Meta's **official Python Business SDK** there.

### Acceptable alternative

If you want one language across app + worker, Meta also maintains an **official Node.js Business SDK**. It is valid, but for this MVP I would still keep the first integration layer as direct REST unless the SDK clearly saves time during the POC.

### Why not make the SDK primary on day one

- The SDK surface is broad and autogenerated.
- It can hide the raw request/response details that matter while debugging permissions and payload validation.
- The MVP only needs a small number of endpoints, so the SDK does not buy much yet.

## Concrete Risks

1. **Token and permission mismatch**
   User tokens only expose accounts/assets the user can access. Meta's official Postman docs also note that managing other people's ad accounts requires **Advanced Access** for `ads_read` and/or `ads_management`.

2. **Asset dependency failures**
   Draft ad creation may require Page, Instagram actor, pixel, custom conversion, promoted object, or other business assets depending on objective.

3. **Schema drift / objective-specific validation**
   Campaign, ad set, creative, and ad payloads are not interchangeable. The generated plan must be converted into a deterministic payload that fits the chosen objective.

4. **Rate limits and query cost**
   Meta's official docs and official SDK materials point to rate limiting and recommend care with batching and large insights pulls. Insights also has async jobs for large result sets.

5. **Pagination**
   List endpoints paginate. Tool code must handle cursors or deliberately cap results for MVP.

6. **App secret proof**
   Meta's official SDK docs recommend enabling App Secret Proof for server API calls. If enabled in the app, requests must include it.

7. **Unsafe abstraction through MCP**
   A generic MCP tool layer makes it easier for agent code to request broad write actions unless we add another policy layer on top.

## Recommended Weekend Scope For Meta Tools

Ship only these deterministic tools first:

- `meta_list_ad_accounts`
- `meta_get_account_summary`
- `meta_get_insights`
- `meta_create_paused_campaign_draft`
- `meta_create_paused_adset_draft`
- `meta_create_adcreative_draft`
- `meta_create_ad_draft`

Optional only after the above works:

- `meta_list_pages`
- `meta_list_instagram_accounts`
- `meta_list_pixels`

Do **not** ship these in the first pass:

- update live campaign status
- budget edits on existing campaigns
- audience upload / lookalike automation
- broad generic `meta_request` tools

## Parent-Agent Decision

Use **direct server-side REST tools in TypeScript** for the MVP and treat Meta's official Postman workspace as the source of truth for endpoint shapes. Keep MCP out of the critical path. Keep official Meta SDKs as a fallback for a later Python worker, not as the first integration choice.
